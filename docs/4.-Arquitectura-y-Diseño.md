## 🔄 Flujo de Comunicación

### Crear Item
```
Main.java
  ↓
new Item(nombre, precio, cantidad)
  ↓
Item.constructor()
  ├─ setNombre()     ← Validación
  ├─ setPrecio()     ← Validación
  └─ setCantidad()   ← Validación
  ↓
Item válido
```

### Agregar al Carrito
```
Main.java
  ↓
carrito.agregarItem(item)
  ↓
CarritoDeCompras.agregarItem()
  ├─ Validar item no nulo
  ├─ Buscar en HashMap
  ├─ Si existe: incrementar cantidad
  └─ Si no existe: agregar nuevo
  ↓
HashMap actualizado
```

### Mostrar Carrito
```
Main.java
  ↓
carrito.imprimirDetalle()
  ↓
CarritoDeCompras.imprimirDetalle()
  ├─ imprimirListaArticulos()
  │   └─ Para cada item: item.toString()
  └─ imprimirTotal()
      └─ calcularTotal()
  ↓
Carrito impreso en consola
```

## 📦 Estructura de Datos

### HashMap (Estructura Principal)

```
HashMap<String, Item>
│
├─ "Camiseta" → Item {
│                ├─ nombre: "Camiseta"
│                ├─ precio: 20.99
│                ├─ cantidad: 2
│                └─ calcularPrecioTotal(): 41.98
│              }
│
├─ "Pantalones" → Item {
│                  ├─ nombre: "Pantalones"
│                  ├─ precio: 34.99
│                  ├─ cantidad: 1
│                  └─ calcularPrecioTotal(): 34.99
│                }
│
└─ "Zapatos" → Item {
                ├─ nombre: "Zapatos"
                ├─ precio: 89.99
                ├─ cantidad: 1
                └─ calcularPrecioTotal(): 89.99
              }
```

**Ventaja:** O(1) para búsqueda, inserción y eliminación

## 🔐 Encapsulación

### Item.java (Privacidad Completa)
```
private String nombre        ← No accesible directamente
private double precio        ← No accesible directamente
private int cantidad         ← No accesible directamente

public String getNombre()    ← Acceso controlado
public double getPrecio()    ← Acceso controlado
public int getCantidad()     ← Acceso controlado

public void setNombre(...)   ← Modificación con validación
public void setPrecio(...)   ← Modificación con validación
public void setCantidad(...) ← Modificación con validación
```

**Beneficio:** Imposible estado inválido

```

    ├─ Intentar operación
    ├─ Capturar excepciones
```

## 🎨 Principios de Diseño Aplicados

### 1. Separación de Responsabilidades
```
Item        → Responsabilidad: Datos y validación
Carrito     → Responsabilidad: Gestión de colección
Main        → Responsabilidad: Presentación
```

### 2. Encapsulación
```
Atributos privados
Getter/setter para acceso controlado
Validación en el setter
```

### 3. Bajo Acoplamiento
```
Main no conoce detalles de Item
Carrito usa Item solo a través de getters
Item no conoce que está en un Carrito
```

### 4. Escalabilidad
```
HashMap O(1) en lugar de ArrayList O(n)
Fácil agregar nuevas operaciones
Fácil cambiar implementación interna
```

### 5. Mantenibilidad
```
Código limpio y legible
Métodos pequeños y enfocados
Documentación completa
```

## 🎯 Decisiones de Diseño Clave

### Decisión 1: HashMap en lugar de ArrayList
**Por qué:** O(1) en búsquedas vs O(n)  
**Impacto:** 1000x más rápido con millones de items  

---

### Decisión 2: Validación en setters
**Por qué:** Una sola fuente de verdad  
**Impacto:** Imposible estado inválido  

---

### Decisión 3: toString() en Item
**Por qué:** Item controla su representación  
**Impacto:** Bajo acoplamiento entre clases  

---

### Decisión 4: Excepciones explícitas
**Por qué:** Errores detectados inmediatamente  
**Impacto:** Debugging fácil, sistemas robustos  

---

### Decisión 5: Métodos pequeños
**Por qué:** Una responsabilidad por método  
**Impacto:** Código reutilizable y testeable

## 🔄 Ciclo de Vida de un Item

```
1. Creación
   new Item(nombre, precio, cantidad)
        ↓
2. Validación (en setters)
   Nombre: ✓ No vacío
   Precio: ✓ En rango
   Cantidad: ✓ > 0
        ↓
3. Almacenamiento
   Se agrega al HashMap del carrito
        ↓
4. Uso
   - Leer datos (getters)
   - Calcular precio total
   - Formatear para impresión
        ↓
5. Modificación
   - Actualizar cantidad
   - Validación ocurre en setter
        ↓
6. Eliminación
   - Remove del HashMap
   - Si hay referencias: garbage collection
```

## 🧪 Testabilidad de la Arquitectura

### Item es Testeable
```
// Crear item
Item item = new Item("Test", 10.00, 5);

// Probar métodos
assert item.getNombre().equals("Test");
assert item.getPrecio() == 10.00;
assert item.getCantidad() == 5;
assert item.calcularPrecioTotal() == 50.00;
```

### Carrito es Testeable
```
// Crear carrito
CarritoDeCompras carrito = new CarritoDeCompras();

// Probar operaciones
carrito.agregarItem(item);
assert carrito.obtenerNumeroArticulos() == 1;
assert carrito.calcularTotal() == 50.00;
```

## 🎓 Lecciones de Arquitectura

1. **Separación clara** facilita mantenimiento
2. **Encapsulación fuerte** previene errores
3. **Interfaces simples** mejoran usabilidad
4. **Validación en capas** es defensa en profundidad
5. **Bajo acoplamiento** permite cambios sin ruptura
